Scope clarification

What this spec covers — and what it doesn't

The only custom UI to build is the admin panel — the product owner's screens for managing tenants and their flows. Customers interact through WhatsApp and Instagram's own native apps; no chat UI is needed from us.

✓ In scope — build this
  • Login screen (FR-01)
  • Dashboard / overview
  • Tenant list + create/edit (FR-15, FR-16)
  • Per-client config: greeting, closing, handoff toggle
  • Flow builder — steps, buttons, linking (FR-17)
  • Conversation log viewer (FR-18)
✗ Out of scope — platform handles it
  • Any chat / messaging UI
  • WhatsApp / Instagram client rendering
  • Phone frames or simulated device views
  • Customer-facing onboarding
  • Agent / support inbox (handoff destination)

Appears over the Clients list. Centred, scrim behind. Dismissable by clicking scrim or Cancel. Create button disabled until Business name has ≥ 2 chars.

Fields

Required, ≥ 2 chars. Used for greeting auto-fill: "Hi 👋 Thanks for messaging {name}!"

Optional at creation. E.164 format validated on save. Displayed in table.
Instagram account
Optional. Validated as @handle pattern.
Handoff email
Optional. If provided, handoff_enabled defaults to true.

Clicking scrim or Cancel closes without saving. No confirmation needed (nothing was committed).

Screen 5 — most important
5

Flow builder FR-17

The core authoring surface. Step list with inline expand-to-edit. Each step = one bot message + N button options, each linking to the next step or ending the flow.

5a — Collapsed view (browse mode)

Flow builder — all steps collapsed, valid flow, Active badge
✓
Validation banner
Top bar shows flow health: green "Flow valid" badge + plain-text confirmation when all checks pass. Red "N issues" badge + error list below when checks fail. Activate/Deactivate button at right — disabled when flow is invalid and inactive.
A
Step card (collapsed)
Shows: monospace step ID, step label, START / END badge if applicable, option count chip, option map (button label → target step name). Clicking anywhere on the header expands to edit mode.
B
Option map
Visible when collapsed. Each row: [Button label chip] › [Target step name] or "End flow" in italic. Gives the owner a quick read of the branching logic without opening the editor.
C
Add step
Ghost button below the step list. Creates a new step with a placeholder message, no options, not marked as start. Automatically opens in edit mode.

5b — Expanded / edit mode (one step open at a time)

Step editor — name, message, start toggle, buttons with targets
A
Step name
Internal label, not shown to customers. Any string. Used in the option map targets of other steps.
B
Message text
Sent to the customer when this step is reached. Free text, supports emoji. WhatsApp recommends ≤ 1024 chars; show character count if over 800.
C
Start step toggle
Exactly one step must be the start. Toggling ON automatically toggles OFF any previously marked start step — no multi-select. Toggling OFF the only start step → validation error immediately.
D
Button / option rows
Each row: [Label input 20/20] › [Target step select] [🗑]. Label max 20 chars — counter shown. Target dropdown lists all other steps + "⏹ End flow". "Add option" button appends a new row.
!
> 3 options warning
WhatsApp supports max 3 interactive buttons in a single message. If a step has 4+ options, show inline warning: "4+ options → sent as a list message." Still allowed — just flagged.
✕
Delete step
Red "Delete step" button at bottom-left of edit panel. No modal confirmation — but any other steps whose options pointed to this step have their next set to null (End flow). This may create new validation errors — shown immediately.

5d — Validation rules (blocks activation)

Rule IDNameCondition that triggers the error
V-01No start stepZero steps have isStart = true. Cannot activate or even send a greeting.
V-02Multiple start stepsMore than one step has isStart = true. Should never occur if toggle logic is correct; guard against it anyway.
V-03Broken linkAn option's next references a step ID that no longer exists (e.g. step was deleted after the link was made).
V-04Unreachable stepA step exists but no path from the start step can reach it. It will never be shown to a customer. Still saves; just can't activate.
V-05 (soft)Button label too longLabel > 20 chars — input is hard-capped at 20 so this only fires if data was injected. Show as warning, not a blocker.
V-06 (soft)Too many buttons> 3 options on a step — sent as list message instead of buttons. Warn inline on the step, don't block activation.

Global patterns

States & feedback patterns

Toast notifications

Bottom-centre toasts for async confirmations. Auto-dismiss after 2.6s. Dark pill with check icon. Used for: "Client created", "Conversation logged", "Handoff email sent".

Loading states

Button spinner (replace icon + text with spinner, disable button). Skeleton rows for table/list loads. No full-page spinners except initial auth check.

Empty states

Each list / table has a centred empty state with an icon, one-line label, and a CTA where applicable (e.g. "Add your first client"). No blank boxes.

Inline validation (flow builder)

Validation runs on every change — no explicit "Validate" button. Errors surface as a red card above the step list and a red badge in the top banner. Errors clear automatically when resolved.
